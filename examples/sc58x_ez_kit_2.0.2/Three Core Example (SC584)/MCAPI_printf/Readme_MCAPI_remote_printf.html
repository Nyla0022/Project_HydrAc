1. Description

This example uses MCAPI to send text logging and shutdown commands from node 1 to node 0. Each node corresponds to a core in the processor.

2. Processor

  • ADSP-SC5xx

3. Tested with

  • ADSP-SC589 EZ-Board
  • ADSP-SC573 EZ-Board
  • Analog Devices CrossCore Embedded Studio
  • ADDS ICE-1000

4. Overview

This example uses the MCAPI inter-core communication mechanism to allow node 1 (normally core 1) to send logging or error strings to an output stream which is managed by node 0 (normally core 0). This stream might also be used by node 0 for its own logging (though not in this example) thus providing a unified log of activity on both cores. The example writes the logging stream to stdout (the console) but this could be modified to go to a file or to a UART or similar. An additional facility is provided which allows node 1 to instruct node 0 to shut down (exit the program) after logging a final message. This is used to terminate the execution of the example. The "server" on node 0 is organized as a framework which allows additional MCAPI-based services to easily be added, and processed as part of a main loop which can also be performing application work. The "client" on node 1 uses functions providing a printf-style interface, with a standard format string and argument list. These functions take care of the details of formatting the text and sending it via MCAPI to the server on the other core.

5. User Configuration Macros

An alternative implementation of the RemotePrintf() client function is provided as part of the source file RemotePrintf.c. This version uses blocking rather than non-blocking MCAPI functions, and may be selected by uncommenting #define BLOCKING in the header file RemotePrintf.h. It is simpler than the default implementation, but less efficient for use in non-multithreaded applications.
